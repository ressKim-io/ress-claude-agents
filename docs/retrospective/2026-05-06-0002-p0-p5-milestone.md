## 무엇이 잘 됐나

### 1. Plan 본문 + Decision Log SSOT가 세션 끊김에 강했다
P0에서 plan 본문(`0002-standardization-and-control-plane.md`)을 영구 박았고, 매 phase마다 progress.md Decision Log에 1줄씩 박는 규율을 유지. 세션이 끊겨도 `cat docs/migration/0002-progress.md`만으로 다음 phase 진입점 + 결정 컨텍스트가 즉시 회복됨. 결과: P3까지 진행 후 별도 세션에서 P4 진입할 때 비용 거의 0.

### 2. Spike → 결정 → 구현 분리가 over-engineering 회피
P4-A spike(30분)에서 ".codex/agents/*.toml 변환 로직 소스 미식별" 위험을 해소하면서 정책 5건을 박음. 그중 #4("`tomlify` 미도입, 직접 stringify 50줄")는 의존성 추가 없이 끝낼 수 있음을 증명 — runtime deps 4/5 유지. 비슷하게 P5에서 picomatch 추가 회피(50줄 inline glob→regex).

### 3. 의미 round-trip 게이트가 byte-exact의 fragility를 회피
P4-A 결정 #3 초기엔 "기존 46개 toml과 byte-exact diff 0 게이트"였으나 Step B 진입 직후 TOML 멀티라인 첫 `\n` 무시 규칙 + yaml 인용 차이 + trailing newline 변동으로 fragile하다고 판단해 **의미 일치(parse→data 동일)**로 정정. 결정 정정도 Decision Log에 박아 추적 가능.

### 4. 9 키 보존 옵션 매트릭스 평가가 단순화의 전형
P4-B에서 codex skills toml에 SKILL.md frontmatter 9 키를 어떻게 보존할지 3 옵션(A: TOML table section / B: body header prepend / C: `manifest_yaml=''…''` 단일 string) 비교. C 채택 — 구현 20줄, 손실 0, P5 hook이 yaml parse 즉시 사용. "TOML-native 풀어쓰기"는 P6 pilot에서 재결정으로 미룸.

### 5. Dual-tree 충돌의 두 옵션 비교 후 더 단순한 쪽
P4-E에서 `.claude/skills/<cat>/<n>/SKILL.md` 디렉토리 형식이 lint+inventory와 충돌. 옵션 A(lint+inventory ~30줄 갱신) vs 옵션 B(.gitignore + CI drift는 codex+cursor만). plan §86-89 정신과 정합하는 B 채택 — 코드 변경 ~10줄로 끝.

### 6. 사용자 의향 반영 후 단계 압축
P4 진행 중 사용자가 "쓸데없이 단계 많은 거 아니지?" 지적. 즉시 trade-off 솔직 점검(spike A 필수 / B 단독 / C-D-E 묶음 가능) 후 1 commit으로 묶지 않고 4 commit으로 나눔. P5에서 같은 패턴 응용 — 3 commit으로 더 압축. 매 phase의 변경 폭에 맞춤.

## 무엇이 어려웠나 / 실수

### 1. .gitignore에 `.codex/` 박혀있던 것 P4-C까지 발견 못함
P0 plan §86-89는 ".codex/agents/는 commit 유지" 명시인데 기존 `.gitignore` L34 `.codex/`로 46 .toml + 새 P4 산출물이 모두 untracked. P4-C 진행 중 git status로 발견 → unignore 결정. 더 일찍 진단했어야 함 — P0 plan 작성 시점에 .gitignore 검수 필요.

### 2. CLI parser 첫 구현이 `--flag=value` 미지원
첫 P4 view generation 시 `--tool=claude`로 호출하다 "unknown adapter flag" 오류. 즉시 보강했지만 P3 lint subcommand 시점부터 파서 컨벤션을 GNU long-opt 표준에 맞췄어야 했음. 6번째 subcommand(admit)에선 처음부터 `=` 형식 지원.

### 3. lint "skill name == basename" 룰이 SKILL.md 디렉토리 형식을 깨뜨림
P4 첫 generate 후 `validate-skill-frontmatter.sh` 10건 fail. SKILL.md 형식은 P0 plan 명시 표준인데 lint가 미대응. 사용자가 "단계 많다" 지적하지 않았으면 lint 수정으로 갔을 가능성 — 결국 dual-tree 충돌 회피로 더 단순한 .gitignore 길로 가는 게 정답.

### 4. byte-exact round-trip 잘못된 가정
P4-A 결정 #3 첫 형식이 "byte-exact diff 0". 실제 구현 시 fragile한 게 명확해 P4-B에서 정정. spike에서 Implementation 한 step 미리 시뮬했어야 catch 가능. 정정 자체는 Decision Log에 박혀 추적 가능하니 큰 손해는 X.

### 5. retrospective 디렉토리 부재 (현재 글 작성하면서 발견)
`docs/` 하위 11종 문서 디렉토리(adr/load-test/migration/postmortem/dx/finops/onboarding/retrospective 등)가 CLAUDE.md documentation rule에 정의되어 있는데 `docs/retrospective/` 부재. 본 글이 첫 retrospective. 향후 P6/P7/P8 마일스톤별 회고를 같은 디렉토리에.

## 다음 사이클(P6~P8)에 가져갈 lessons

### 단계 압축의 적정선
- spike(코드 X, 결정만) = 단독 commit X, dev-log narrative만으로 충분
- 핵심 구현(adapter.ts/admit.ts 같은 신규 파일) = 단독 commit
- wiring(init.ts step N) = 1 commit
- CI/lint/gitignore = wiring과 묶음 가능 (작으면)
- docs(progress + dev-log) = 단독 commit (회고/검색성)

P4 = 4 commit, P5 = 3 commit이 sweet spot. **commit 수는 변경 의미 단위에 맞추되 사용자 ROI 최우선**.

### Decision Log 1줄 규율
52건 박음 — 이게 다음 sub-decision 비용을 낮춤. 미래 누군가 "왜 manifest_yaml 형태로 했나" 물으면 P4-B 한 줄로 컨텍스트 회복. 향후 P6+에서도 매 sub-decision 1줄 박는 규율 유지.

### dev-log narrative의 가치
spike + step별 narrative 7건. 코드만으로는 "왜 이 길로 갔나"를 회복 불가. 회고/문제 분석 시 dev-log가 SSOT. 다음 사이클도 phase 종결마다 dev-log 1건 작성 유지.

### lint/gitignore는 plan 단계에서 미리 점검
1번 실수의 교훈. P6 진입 시 plan 명시(`.claude/skills/*` 디렉토리 형식 적용 범위)와 lint/gitignore 정합성을 사전 검수. 새 파일 형식 도입 시 기존 자동화 5종(lint scripts)에 미치는 영향을 plan 단계에서 표로 정리.

### 의존성 정책의 힘
runtime deps ≤ 5 정책이 "tomlify 미도입" / "picomatch 미도입" 결정을 추진. 직접 구현 50줄로 끝나는 일을 lib 추가로 해결하는 유혹 차단. P6+에서도 정책 유지 — 새 subcommand 추가 시 의존성 1개 증가는 별도 ADR.
